# オブジェクトストリーム仕様 (Object Streams)

> **ISO 32000-1:2008 &sect;7.5.7** / **ISO 32000-2:2020 &sect;7.5.7**

## 1. 概要

オブジェクトストリーム（ObjStm）は PDF 1.5 で導入された構造圧縮メカニズムである。複数のインダイレクトオブジェクトを1つのストリームオブジェクトに集約し、ファイルサイズを削減する。

従来の PDF では各インダイレクトオブジェクトが `obj` / `endobj` キーワードで囲まれて個別に存在するが、ObjStm では複数オブジェクトが1つのストリーム内にパックされる。XRef ストリーム（Type=2 エントリ）と組み合わせて使用される。

### 1-1. 導入バージョン

| バージョン | 仕様 | 変更内容 |
|:-----------|:-----|:---------|
| PDF 1.5 | ISO 32000-1:2008 &sect;7.5.7 | ObjStm 導入 |
| PDF 2.0 | ISO 32000-2:2020 &sect;7.5.7 | 変更なし（仕様継続） |

## 2. ObjStm 辞書エントリ

ObjStm はストリームオブジェクトであり、ストリーム辞書に以下のエントリを持つ。

| キー | 型 | 必須 | 説明 |
|:-----|:---|:-----|:-----|
| `/Type` | name | 必須 | 値は `/ObjStm`（固定） |
| `/N` | integer | 必須 | ストリーム内に格納されたオブジェクトの数 |
| `/First` | integer | 必須 | オブジェクトデータ領域の開始バイトオフセット（展開後データ内での位置） |
| `/Filter` | name or array | 任意 | 圧縮フィルタ。省略時はストリームデータが未圧縮であることを意味する |
| `/DecodeParms` | dictionary or array | 任意 | フィルタパラメータ（Predictor 等） |
| `/Extends` | indirect-ref | 任意 | 拡張元の ObjStm への間接参照 |
| `/Length` | integer | 必須 | ストリームデータのバイト長（ストリーム共通エントリ） |

### 2-1. `/Filter` について

ObjStm のストリームデータは通常 `/FlateDecode`（zlib）で圧縮される。`/Filter` が省略された場合、ストリームデータは未圧縮である。

```
% 圧縮された ObjStm
15 0 obj
<< /Type /ObjStm
   /N 3
   /First 24
   /Filter /FlateDecode
   /Length 200
>>
stream
... (zlib圧縮データ) ...
endstream
endobj
```

```
% 未圧縮の ObjStm
15 0 obj
<< /Type /ObjStm
   /N 2
   /First 12
   /Length 50
>>
stream
10 0 11 15 << /Key /Value >> true
endstream
endobj
```

## 3. 内部バイナリ構造

ObjStm の展開後データは2つの領域に分かれる。

```
|<--- /First バイト --->|<--- オブジェクトデータ領域 --->|
|   オフセットテーブル     |   オブジェクト本体            |
```

### 3-1. オフセットテーブル領域

展開後データの先頭 `/First` バイトに、`/N` 組の整数ペアが空白区切りで格納される。

```
objNum1 offset1 objNum2 offset2 ... objNumN offsetN
```

- `objNum`: オブジェクト番号（10進整数）
- `offset`: `/First` からの相対バイトオフセット（10進整数）

例: `/N=3`, `/First=24` の場合

```
10 0 11 15 12 30
```

| インデックス | objNum | offset | データ開始位置 |
|:-------------|:-------|:-------|:---------------|
| 0 | 10 | 0 | First + 0 = 24 |
| 1 | 11 | 15 | First + 15 = 39 |
| 2 | 12 | 30 | First + 30 = 54 |

### 3-2. オブジェクトデータ領域

`/First` 以降に、各オブジェクトの値が直接記述される。

- `obj` / `endobj` キーワードは**使用しない**
- オブジェクト間の境界はオフセットテーブルのオフセット値で決定される
- 各オブジェクトは辞書、配列、数値、文字列、名前、真偽値、null のいずれか

```
先頭24バイト: "10 0 11 15 12 30"（オフセットテーブル）
24バイト目〜: << /Key1 /Value1 >>    (obj 10)
39バイト目〜: << /Key2 /Value2 >>    (obj 11)
54バイト目〜: true                   (obj 12)
```

## 4. 制約事項

ISO 32000-1:2008 &sect;7.5.7 では以下の制約が定められている。

### 4-1. ストリームオブジェクト格納不可

ObjStm 内にストリームオブジェクト（`stream` / `endstream` を持つオブジェクト）を格納することはできない。ObjStm 内のオブジェクトには `obj` / `endobj` キーワードがないため、ストリームの境界を示す手段がない。

### 4-2. 世代番号は常に 0

ObjStm 内に格納されたオブジェクトの世代番号は常に 0 である。世代番号が 0 でないオブジェクトは ObjStm に格納できない。

### 4-3. ObjStm の入れ子禁止

ObjStm 自身を別の ObjStm 内に格納することはできない。ObjStm はストリームオブジェクトであるため、制約 4-1 により自動的に禁止される。

### 4-4. 暗号化

暗号化ドキュメントでは、ObjStm 内の個々のオブジェクトは個別に暗号化されない。ObjStm のストリームデータ全体が1つの単位として暗号化される。

### 4-5. 格納不可オブジェクト

以下のオブジェクトは ObjStm に格納してはならない:
- 暗号化辞書のオブジェクト
- `/Length` エントリが間接参照であるストリームの `/Length` オブジェクト
- ドキュメントカタログ、ページツリーのルートノードなど、xref から直接参照される必要があるオブジェクト（実装依存）

## 5. XRef との連携

### 5-1. XRef ストリームの Type=2 エントリ

XRef ストリーム（ISO 32000-1:2008 &sect;7.5.8）の各エントリは 3 つのフィールドを持つ。Type=2 エントリは ObjStm 内のオブジェクトを参照する。

| フィールド | 意味 |
|:-----------|:-----|
| Field1 (type) | `2` — オブジェクトストリーム内のオブジェクト |
| Field2 | ObjStm のオブジェクト番号 |
| Field3 | ストリーム内インデックス（0-based） |

例: オブジェクト 10 が ObjStm 15 のインデックス 0 に格納されている場合

```
Field1=2, Field2=15, Field3=0
```

### 5-2. テキスト xref との関係

テキスト形式の xref テーブル（`xref` キーワードで始まる従来形式）からは ObjStm 内のオブジェクトを参照できない。ObjStm は XRef ストリームを使用する PDF でのみ利用可能である。

## 6. /Extends による拡張

`/Extends` エントリにより、別の ObjStm を拡張できる。インクリメンタルアップデートで ObjStm にオブジェクトを追加・上書きする場合に使用される。

```
20 0 obj
<< /Type /ObjStm
   /N 1
   /First 6
   /Extends 15 0 R
>>
stream
13 0 << /NewKey /NewValue >>
endstream
endobj
```

リーダーは `/Extends` チェーンを走査し、同じオブジェクト番号が複数の ObjStm に存在する場合は最も新しい（チェーンの先頭に近い）ものを使用する。

## 7. パース手順（概念的フロー）

ObjStm からオブジェクトを抽出する一般的な手順:

1. XRef から Type=2 エントリを検出し、ObjStm のオブジェクト番号とインデックスを取得
2. ObjStm オブジェクトを通常のインダイレクトオブジェクトとして解決
3. ストリーム辞書を検証（`/Type`=`/ObjStm`, `/N`, `/First` の存在と型）
4. `/Filter` に基づきストリームデータを展開（`/FlateDecode` → zlib展開、省略 → そのまま使用）
5. 展開後データの先頭 `/First` バイトをオフセットテーブルとしてパース
6. オフセットテーブルから `/N` 組の `(objNum, offset)` ペアを取得
7. 指定インデックスの `offset` を使い、`/First + offset` の位置からオブジェクトをパース
8. パースしたオブジェクトのオブジェクト番号がオフセットテーブルの `objNum` と一致することを検証

### 7-1. バイナリレイアウト図

```
ObjStm ストリームデータ (展開後)
+------------------------------------------------------------------+
| オフセットテーブル (/First バイト)                                   |
| "10 0 11 15 12 30"                                               |
+------------------------------------------------------------------+
| /First + 0:  << /Type /Font /Subtype /Type1 >>     (obj 10)     |
| /First + 15: << /Length 100 >>                       (obj 11)     |
| /First + 30: true                                    (obj 12)     |
+------------------------------------------------------------------+
```

## 8. エッジケースと寛容処理

| ケース | PDF仕様上の扱い | 推奨処理 |
|:-------|:----------------|:---------|
| `/First` や `/N` が不正（負値、実数、不在） | 不正な PDF | エラーを返す |
| オフセットテーブルのペア数が `/N` と不一致 | 不正な PDF | エラーを返す |
| 未圧縮 ObjStm（`/Filter` 省略） | 合法（ただし稀） | 展開せずそのまま使用 |
| `/DecodeParms` 付き `/FlateDecode` | 合法（Predictor 逆変換が必要） | 対応するか、非対応としてエラーを返す |
| ObjStm 内にストリームオブジェクトが存在 | 仕様違反 | エラーを返す |
| `/Extends` チェーンが循環 | 不正な PDF | 循環検出してエラーを返す |
| オフセットが展開後データ範囲外 | 不正な PDF | エラーを返す |
| オフセットテーブル内の `objNum` が XRef の `objNum` と不一致 | 破損 PDF の可能性 | エラーを返すか、寛容に処理する |
