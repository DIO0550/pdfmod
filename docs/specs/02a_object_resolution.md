# 02a. インダイレクトオブジェクト解決

## 1. 概要

PDFファイル内のオブジェクトは、他のオブジェクトを**間接参照（Indirect Reference）**を通じて参照する。間接参照は `N G R` の形式で表現され、Nはオブジェクト番号、Gは世代番号である。

間接参照を実体のPDFオブジェクトに変換するプロセスを**オブジェクト解決（Object Resolution）**と呼ぶ。適合リーダーは、相互参照テーブル（xref）を用いてオブジェクトの物理的な位置を特定し、その位置からオブジェクトを読み込む。

```
間接参照: 5 0 R
    ↓ xref テーブルで位置を特定
    ↓ 該当オフセットからオブジェクトを読み込み
実体: << /Type /Page /MediaBox [0 0 612 792] >>
```

## 2. xrefテーブルによるオブジェクト位置の特定

相互参照テーブルは、各オブジェクトの物理的な位置をインデックスとして保持する。エントリは以下の3タイプに分類される。

### 2.1 Free Entry (type=0)

削除されたオブジェクトを示す。フリーリストとしてリンクリスト構造を形成する。

```
xref
0 6
0000000000 65535 f    ← オブジェクト0: フリーリストの先頭
0000000010 00000 n
0000000000 00001 f    ← オブジェクト2: 削除済み（世代番号+1）
```

適合リーダーは、Free Entryに対する間接参照を解決する際に**nullオブジェクト**を返すべきである。

### 2.2 Used Entry (type=1)

通常のオブジェクト。ファイル内のバイトオフセットで位置が指定される。

```
xref
0 3
0000000000 65535 f
0000000010 00000 n    ← オブジェクト1: オフセット10、世代0
0000000512 00000 n    ← オブジェクト2: オフセット512、世代0
```

一般的なPDFリーダーは、指定されたオフセットにシークし、`N G obj ... endobj` 構文を解析してオブジェクトを読み込む。

ただし、**本ライブラリの現状の `ObjectResolver` 実装では type=1 の解決は未実装**であり、このケースではオブジェクトを読み込まず**エラーを返す**。

### 2.3 Compressed Entry (type=2)

オブジェクトストリーム（ObjStm）内に格納されたオブジェクト。xrefストリーム形式でのみ使用される。

| フィールド | 内容 |
|:-----------|:-----|
| type | 2 |
| streamObject | 親ストリームのオブジェクト番号 |
| indexInStream | ストリーム内のインデックス |

Compressed Entryの世代番号は常に0である。

### 2.4 解決フロー

```
間接参照 (N G R) を受け取る
    ↓
xref テーブルで N を検索
    ↓
┌─ 未登録 ────────────→ null を返却
├─ type=0 (Free) ─────→ null を返却
├─ type=1 (Used) ─────→ オフセットにシーク → obj...endobj を解析
└─ type=2 (Compressed)→ 親 ObjStm を解決 → ストリーム内のオブジェクトを抽出
```

## 3. 間接オブジェクト定義の構文

type=1（Used Entry）で参照されるオブジェクトは、以下の構文で定義される。

```
N G obj
  <オブジェクト本体>
endobj
```

- **N**: オブジェクト番号（正の整数）
- **G**: 世代番号（非負整数、通常は0）
- **obj / endobj**: 開始・終了キーワード

### 3.1 ヘッダ検証

適合リーダーは、xrefエントリのオフセットに `N G obj` が存在することを検証すべきである。オブジェクト番号と世代番号がxrefエントリと一致しない場合、ファイルが破損している可能性がある。

## 4. 世代番号の役割

世代番号は、オブジェクトの削除と再利用を管理するメカニズムである。

### 4.1 ライフサイクル

1. 新規オブジェクト → 世代番号 0
2. インクリメンタルアップデートで削除 → フリーリストに追加、世代番号+1
3. 同じオブジェクト番号で再利用 → 新しい世代番号で使用

### 4.2 世代番号の不一致

間接参照 `N G R` の世代番号Gと、xrefエントリの世代番号が一致しない場合、その参照は無効とみなされる。適合リーダーは**nullオブジェクト**を返すべきである。

最大世代番号は **65535** である（ISO 32000-1 §7.5.4）。

## 5. オブジェクトストリーム内のオブジェクト

### 5.1 ObjStm の構造

オブジェクトストリーム（`/Type /ObjStm`）は、複数のオブジェクトを単一のストリーム内に格納するコンテナである。

```
10 0 obj
<< /Type /ObjStm /N 3 /First 24 >>
stream
11 0 12 100 13 200    ← ヘッダ: オブジェクト番号とオフセットのペア
<< /Key1 /Value1 >>  ← オブジェクト11のデータ
<< /Key2 /Value2 >>  ← オブジェクト12のデータ
<< /Key3 /Value3 >>  ← オブジェクト13のデータ
endstream
endobj
```

| キー | 必須 | 内容 |
|:-----|:-----|:-----|
| /Type | 必須 | `/ObjStm` |
| /N | 必須 | 格納されているオブジェクト数 |
| /First | 必須 | 最初のオブジェクトデータへのバイトオフセット |

### 5.2 世代番号の制約

ObjStm内のオブジェクトの世代番号は**常に0**である。世代番号が0でないオブジェクトをObjStm内に格納することはできない。

### 5.3 格納制限

以下のオブジェクトはObjStm内に格納できない:
- ストリームオブジェクト自身
- 世代番号が0でないオブジェクト
- 暗号化辞書
- xrefストリーム内の `/Length` エントリの値が間接参照のオブジェクト

## 6. 循環参照

### 6.1 仕様上の禁止

PDF仕様は、間接参照による循環参照を明示的に禁止してはいないが、適合ファイルにおいて意味のある循環参照は発生しない。ただし、破損ファイルや悪意のあるファイルで循環参照が存在する可能性がある。

### 6.2 適合リーダーの対応

適合リーダーは循環参照を検出し、無限ループを回避すべきである。検出手法として以下が推奨される:

- **訪問済みセット**: 解決チェーン内で既に訪問したオブジェクトIDを記録し、再訪問を検出
- **深さ制限**: 解決チェーンの最大深度を設定（例: 100段）

循環参照を検出した場合、エラーを返却するか、nullオブジェクトとして処理する。

## 7. 解決結果のキャッシュ

### 7.1 キャッシュの必要性

PDF内では同一のオブジェクトが多数の箇所から参照されることが一般的である（例: フォント辞書、カラースペース）。解決結果をキャッシュすることで、同一オブジェクトの重複パースを回避し、パフォーマンスを大幅に改善できる。

### 7.2 キャッシュキー

キャッシュキーは `"オブジェクト番号-世代番号"` の形式を推奨する。インクリメンタルアップデートにより、同一オブジェクト番号でも異なる世代番号のオブジェクトが存在しうるためである。

### 7.3 LRU戦略

メモリ使用量を制限するため、LRU（Least Recently Used）キャッシュが推奨される。一般的なPDFファイルでは1024エントリ程度の容量で十分なカバレッジが得られる。

## 8. オフセットずれの寛容処理

> **注**: 本章は type=1 (Used Entry) 解決が実装済みであることを前提とした将来の方針である。本ライブラリの現状では type=1 が未実装のため、ここに記載の寛容処理は適用されない。

### 8.1 問題

実世界のPDFファイルでは、xrefテーブルに記録されたバイトオフセットが実際のオブジェクト位置と一致しないケースが存在する。原因として:

- インクリメンタルアップデート時の計算ミス
- ファイル結合ツールのバグ
- 改行コード変換（CR/LF → LF）によるオフセットずれ

### 8.2 対応戦略

1. まず記録されたオフセットに `N G obj` が存在するか確認
2. 存在しない場合、前後数十バイト（例: ±32バイト）を探索
3. それでも見つからない場合はエラーを返却

この寛容処理は、PDF/A等の厳格な適合性が要求されるコンテキストでは無効化すべきである。
