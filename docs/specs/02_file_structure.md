# 02. 物理ファイル構造とランダムアクセス機構

## 1. 概要

有効なPDFファイルは、物理的に以下の4つの主要セグメントから構成される。

```
+-----------------------------+
| ヘッダ (Header)              |  ← ファイル先頭
+-----------------------------+
| ボディ (Body)                |  ← インダイレクトオブジェクト群
|   obj 1 0 ... endobj        |
|   obj 2 0 ... endobj        |
|   ...                       |
+-----------------------------+
| 相互参照テーブル (xref)       |  ← オブジェクト位置のインデックス
+-----------------------------+
| トレイラ (Trailer)           |  ← ドキュメントのエントリーポイント
+-----------------------------+
| %%EOF                       |  ← ファイル終端マーカー
+-----------------------------+
```

**重要**: PDFはファイル末尾から逆方向に解析を開始する。先頭から末尾への順次読み込みによる実装はパフォーマンス上不可能に近い。

## 2. ファイルヘッダ

### 2.1 バージョン宣言

ファイルの先頭に配置される。

```
%PDF-1.7
```

| バージョン | ISO規格 |
|:-----------|:--------|
| %PDF-1.0 〜 %PDF-1.7 | ISO 32000-1 |
| %PDF-2.0 | ISO 32000-2 |

### 2.2 バージョンの上書きルール

ファイルヘッダのバージョン宣言は**絶対的ではない**。ドキュメントカタログ辞書内の `/Version` エントリがヘッダのバージョンを**上書き（オーバーライド）**できる。

```
パーサーの処理順序:
1. ヘッダからバージョンを取得（暫定値）
2. ドキュメントカタログの /Version を確認
3. /Version が存在し、ヘッダより新しい場合は /Version を採用
4. /Version が存在しない場合はヘッダのバージョンを使用
```

#### 実装仕様
```
fn determine_version(header_version: PdfVersion, catalog: &PdfDictionary) -> PdfVersion {
    match catalog.get("/Version") {
        Some(catalog_version) if catalog_version > header_version => catalog_version,
        _ => header_version,
    }
}
```

### 2.3 バイナリファイルインジケータ

ヘッダの直後に配置される。

```
%PDF-1.7
%âãÏÓ
```

- ASCIIコード127を超える非印字文字（高ビット文字）を**4バイト以上**含むコメント行
- FTPなどのファイル転送プロトコルに対して、このファイルがバイナリデータであることを通知
- 改行コードの意図しない変換によるファイル破損を防止

#### 実装仕様
```
書き込み時:
  - ヘッダ直後に "%\xE2\xE3\xCF\xD3\n" を出力

読み込み時:
  - この行は無視してよい（コメントとして処理）
  - ただし存在を確認することでバイナリ安全性を検証可能
```

## 3. ファイルボディ

ボディ部分にはインダイレクトオブジェクトが配置される。オブジェクトの物理的な配置順序は意味を持たず、xrefテーブルによるランダムアクセスで参照される。

```
1 0 obj
<< /Type /Catalog /Pages 2 0 R >>
endobj

2 0 obj
<< /Type /Pages /Kids [3 0 R] /Count 1 >>
endobj

3 0 obj
<< /Type /Page /Parent 2 0 R /MediaBox [0 0 612 792] /Contents 4 0 R >>
endobj

4 0 obj
<< /Length 44 >>
stream
BT /F1 24 Tf 72 720 Td (Hello World) Tj ET
endstream
endobj
```

## 4. 相互参照テーブル（Xrefテーブル）

### 4.1 従来のテキストベースxrefテーブル

```
xref
0 6
0000000000 65535 f
0000000009 00000 n
0000000058 00000 n
0000000115 00000 n
0000000206 00000 n
0000000300 00000 n
```

#### テーブル構造

| 行 | 説明 |
|:---|:-----|
| `xref` | テーブル開始キーワード |
| `0 6` | 最初のオブジェクト番号とエントリ数 |
| 以降の行 | 各オブジェクトのエントリ（固定長20バイト） |

#### エントリフォーマット（厳密に20バイト固定長）

```
nnnnnnnnnn ggggg s EOL
|          |     |  |
|          |     |  +-- 行末 (CR+LF or LF or CR で2バイト)
|          |     +----- 状態フラグ (1バイト: 'n' or 'f')
|          +----------- 世代番号 (5桁、0埋め)
+---------------------- バイトオフセット (10桁、0埋め)
```

| フィールド | バイト数 | 説明 |
|:-----------|:---------|:-----|
| オフセット | 10 | ファイル先頭からの正確なバイト数 |
| スペース | 1 | 区切り |
| 世代番号 | 5 | オブジェクトの世代（通常 `00000`） |
| スペース | 1 | 区切り |
| 状態フラグ | 1 | `n`（使用中）/ `f`（空き） |
| EOL | 2 | 行末文字 |

#### 特殊ルール
- **オブジェクト番号0のエントリ**: 常に世代番号 `65535` の空き（f）オブジェクト
- オブジェクト番号0は空きオブジェクト連結リストのヘッドとして機能
- 空きオブジェクトのオフセットフィールドは、次の空きオブジェクトの番号を示す

#### 複数サブセクション

xrefテーブルは連続しないオブジェクト番号範囲を持つ複数のサブセクションで構成できる。

```
xref
0 3
0000000000 65535 f
0000000009 00000 n
0000000058 00000 n
10 2
0000000300 00000 n
0000000450 00000 n
```

この例では、オブジェクト0〜2と10〜11のエントリが定義されている。

### 4.2 XRefストリーム（PDF 1.5以降）

プレーンテキストのxrefテーブルの代わりに、**ストリームオブジェクト内のバイナリデータ**として相互参照データを格納する形式。

```
15 0 obj
<< /Type /XRef
   /Size 20
   /W [1 3 1]
   /Root 1 0 R
   /Filter /FlateDecode
   /Length 120
>>
stream
... (Flate圧縮されたバイナリデータ) ...
endstream
endobj
```

#### XRefストリーム辞書の必須エントリ

| キー | 型 | 説明 |
|:-----|:---|:-----|
| `/Type` | 名前 | 必ず `/XRef` |
| `/Size` | 整数 | テーブル内の最大オブジェクト番号 + 1 |
| `/W` | 配列 | 各フィールドのバイト幅を示す配列 [type_width, field2_width, field3_width] |

#### /W 配列によるフィールド定義

`/W [w1 w2 w3]` は各エントリが `w1 + w2 + w3` バイトで構成されることを示す。

| フィールド | 説明 |
|:-----------|:-----|
| Type (w1バイト) | エントリの種類: 0=空き, 1=通常, 2=オブジェクトストリーム内 |
| Field2 (w2バイト) | Type=0: 次の空きオブジェクト番号、Type=1: バイトオフセット、Type=2: 親ストリームのオブジェクト番号 |
| Field3 (w3バイト) | Type=0: 再利用時の世代番号、Type=1: 世代番号、Type=2: ストリーム内のインデックス |

#### エントリタイプ詳細

| Type値 | Field2の意味 | Field3の意味 |
|:-------|:-------------|:-------------|
| 0 | 次の空きオブジェクト番号 | 再利用時の世代番号 |
| 1 | オブジェクトのバイトオフセット | 世代番号 |
| 2 | オブジェクトストリームのオブジェクト番号 | ストリーム内のインデックス |

#### 実装仕様
```
struct XRefEntry {
    entry_type: u8,        // 0, 1, or 2
    field2: u64,           // 解釈はentry_typeに依存
    field3: u32,           // 解釈はentry_typeに依存
}

fn parse_xref_stream(w: [usize; 3], data: &[u8]) -> Vec<XRefEntry> {
    let entry_size = w[0] + w[1] + w[2];
    data.chunks(entry_size)
        .map(|chunk| {
            let type_val = read_uint(&chunk[0..w[0]]);
            let field2 = read_uint(&chunk[w[0]..w[0]+w[1]]);
            let field3 = read_uint(&chunk[w[0]+w[1]..]);
            XRefEntry { entry_type: type_val as u8, field2, field3: field3 as u32 }
        })
        .collect()
}
```

### 4.3 テキストxrefとXRefストリームの統合

適合リーダーは**両方の形式をシームレスに解析・結合**できなければならない。

```
抽象化レイヤの設計:

trait XRefTable {
    fn get_offset(&self, obj_num: u32, gen_num: u16) -> Option<ObjectLocation>;
    fn is_free(&self, obj_num: u32) -> bool;
    fn size(&self) -> u32;
}

enum ObjectLocation {
    DirectOffset(u64),                    // ファイル内のバイトオフセット
    InObjectStream { stream_obj: u32, index: u32 }, // オブジェクトストリーム内
}
```

## 5. トレイラ辞書（Trailer Dictionary）

### 5.1 解析の開始手順

```
1. ファイル末尾の %%EOF マーカーを探す
2. %%EOF の上方向にスキャンして startxref を探す
3. startxref の直後の数値がxrefテーブルのバイトオフセット
4. xrefテーブルの直前（従来形式）またはストリーム辞書内にトレイラ情報がある
```

### 5.2 従来形式のトレイラ

```
trailer
<< /Size 6
   /Root 1 0 R
   /Info 5 0 R
   /ID [<abc123> <def456>]
>>
startxref
408
%%EOF
```

### 5.3 トレイラ辞書の主要キー

| キー | 型 | 必須 | 説明 |
|:-----|:---|:-----|:-----|
| `/Root` | 間接参照 | **必須** | ドキュメントカタログへの参照。ドキュメント全体のルート要素 |
| `/Size` | 整数 | **必須** | xrefテーブルのエントリ総数（最大オブジェクト番号 + 1） |
| `/Info` | 間接参照 | 任意 | ドキュメント情報辞書への参照（作成者、作成日時など） |
| `/ID` | 配列 | 条件付き必須 | 2つの16バイト文字列の配列。ファイルの一意識別に使用 |
| `/Prev` | 整数 | 条件付き | 前の（古い）xrefテーブルのバイトオフセット（インクリメンタルアップデート時） |
| `/Encrypt` | 辞書 | 条件付き | 暗号化辞書への参照 |

### 5.4 /ID 配列の詳細

```
/ID [<永続ID> <変更ID>]
```

| 要素 | 説明 |
|:-----|:-----|
| 第1要素（永続ID） | ファイル作成時に生成。以後変更されない |
| 第2要素（変更ID） | ファイルが変更されるたびに再生成される |

- PDF 2.0では `/ID` の記述が**厳格化**されている
- 暗号化されたPDFでは `/ID` は**必須**

### 5.5 解析フローの完全な手順

```
fn parse_pdf(file: &[u8]) -> Result<PdfDocument> {
    // 1. ファイル末尾から %%EOF を探索
    let eof_pos = find_eof_marker(file)?;

    // 2. startxref キーワードを探索
    let startxref_pos = find_startxref(file, eof_pos)?;

    // 3. xrefテーブルのオフセットを取得
    let xref_offset = parse_startxref_value(file, startxref_pos)?;

    // 4. xrefテーブルを読み込み
    let xref_table = parse_xref(file, xref_offset)?;

    // 5. トレイラ辞書を解析
    let trailer = parse_trailer(file, xref_offset)?;

    // 6. /Prev を辿って古いxrefテーブルを再帰的に読み込み
    let merged_xref = merge_xref_chain(file, xref_table, &trailer)?;

    // 7. /Root を解決してドキュメントカタログを取得
    let catalog = resolve_reference(merged_xref, trailer.root_ref)?;

    // 8. ドキュメント構造の構築
    Ok(build_document(catalog, merged_xref)?)
}
```

## 6. %%EOF マーカー

- ファイルの論理的な終端を示す
- **複数の %%EOF が存在可能**（インクリメンタルアップデート時）
- パーサーは**最後の %%EOF** から解析を開始する
- %%EOF の後にゴミデータが存在する場合がある（寛容に処理すべき）

#### 実装上の注意点
```
%%EOF探索のアルゴリズム:
1. ファイル末尾から最大1024バイトの範囲でスキャン
2. "%%EOF" の文字列を後方検索
3. 見つからない場合はエラー（ただし寛容モードでは範囲を拡大）
4. %%EOF の前後のホワイトスペースは許容する
```
