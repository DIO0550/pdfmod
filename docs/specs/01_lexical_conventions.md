# 01. レキシカル規約とプリミティブデータ型

## 1. 概要

PDFファイルの解析における最下層の処理は、入力されるバイトストリームを適切なトークンに分割する**字句解析（レキシカルスキャナ）**の実装である。PDFの構文はPostScriptから強く影響を受けている。

## 2. バイト分類

ファイル内のバイトは以下の3つに厳密に分類される。

### 2.1 ホワイトスペース文字

トークンの区切りとして機能する文字。

| 文字コード | 名称 | 説明 |
|:-----------|:-----|:-----|
| 0x00 | NUL | ヌル文字 |
| 0x09 | HT | 水平タブ |
| 0x0A | LF | 改行（Line Feed） |
| 0x0C | FF | フォームフィード |
| 0x0D | CR | 復帰（Carriage Return） |
| 0x20 | SP | 空白（スペース） |

#### 実装上の注意点
- 改行の表現は **LF単体**、**CR単体**、**CR+LF** の3パターンすべてを等価に扱わなければならない
- CR+LFのペアは1つの改行として処理する（2つの改行として処理してはならない）
- ストリームデータ内のホワイトスペースは区切り文字としてではなく、データの一部として扱う

### 2.2 デリミタ文字

トークンの構造を構築するための特殊文字。

| 文字 | 用途 |
|:-----|:-----|
| `(` `)` | リテラル文字列の開始・終了 |
| `<` `>` | 16進文字列の開始・終了 / 辞書の開始・終了（二重時） |
| `[` `]` | 配列の開始・終了 |
| `{` `}` | 名前ツリーなどで使用（PostScript互換） |
| `/` | 名前オブジェクトの開始 |
| `%` | コメントの開始 |

### 2.3 通常の文字

上記のいずれにも該当しないバイト。トークン（キーワード、数値など）を構成する。

## 3. プリミティブデータ型

パーサーは以下の**9種類のプリミティブデータ型**へ安全にマッピングしなければならない。

### 3.1 Nullオブジェクト

```
null
```

- 値が存在しないことを示すキーワード
- 継承された辞書エントリを明示的に削除（クリア）するために使用
- 解決できない間接参照の結果としても返される

#### 実装仕様
```
型名: PdfNull
内部表現: シングルトンインスタンス
比較: 参照等価
シリアライズ: "null" (4バイト固定)
```

### 3.2 ブール値

```
true
false
```

- 論理値を表すキーワード
- フラグの制御に使用される
- 大文字小文字を区別する（`True` や `FALSE` は無効）

#### 実装仕様
```
型名: PdfBoolean
内部表現: bool
シリアライズ: "true" (4バイト) / "false" (5バイト)
```

### 3.3 数値（整数・実数）

```
整数の例:  123  -45  0  +17
実数の例:  34.5  -3.62  +123.6  .002  4.  -.002
```

#### 制約事項
- **指数表記（例：1.23e4）は仕様上許可されていない**
- パーサーは指数表記に遭遇した場合：
  - 厳格モード: エラーを送出
  - 寛容モード: ヒューリスティックとして許容（実世界のPDFへの対応）
- 整数の範囲: 実装依存だが、最低でも32ビット符号付き整数をサポートすべき
- 実数の精度: IEEE 754倍精度浮動小数点数を推奨

#### 実装仕様
```
型名: PdfInteger, PdfReal
内部表現: i64 / f64
パース規則:
  - 先頭の '+' は許容（正の符号）
  - 先頭の '-' は負の数
  - '.' のみの入力は無効
  - 連続する '.' は無効
```

### 3.4 リテラル文字列

```
(Hello World)
(Strings may contain newlines
and such.)
(Strings may contain balanced parentheses () and
special characters (*!&}^% and so on).)
(The following is an empty string.)
()
```

#### エスケープシーケンス

| シーケンス | 意味 |
|:-----------|:-----|
| `\n` | 改行（LF: 0x0A） |
| `\r` | 復帰（CR: 0x0D） |
| `\t` | 水平タブ（HT: 0x09） |
| `\b` | バックスペース（BS: 0x08） |
| `\f` | フォームフィード（FF: 0x0C） |
| `\(` | 左括弧 |
| `\)` | 右括弧 |
| `\\` | バックスラッシュ |
| `\ddd` | 8進数表記の文字（dddは0〜7の数字、1〜3桁） |

#### 実装上の注意点
- 括弧のネストが**バランスしている**場合、エスケープなしでも使用可能
- パーサーは括弧のネストレベルを追跡する必要がある
- バックスラッシュの後に認識されない文字が続く場合、バックスラッシュは無視される
- 行末のバックスラッシュは「行継続」を意味する（改行は文字列に含まれない）

#### 実装仕様
```
型名: PdfString (Literal)
内部表現: Vec<u8> (バイト列)
パース状態:
  - depth: i32 (括弧のネスト深度)
  - escape: bool (エスケープ処理中フラグ)
  - octal_buffer: [u8; 3] (8進数バッファ)
```

### 3.5 16進文字列

```
<48656C6C6F>           => "Hello"
<48656C6C6F20576F726C64> => "Hello World"
<901FA3>               => 3バイトのバイナリデータ
<901FA>                => <901FA0> として処理（末尾に0が補完）
```

#### 実装上の注意点
- 各文字ペア（2つの16進数字）が1バイトを表す
- 16進数字間のホワイトスペースは無視される
- **奇数桁の場合**: 末尾に `0` が補完される（例: `<ABC>` は `<ABC0>` として解釈）
- 大文字小文字を区別しない（`a`〜`f` と `A`〜`F` は等価）

#### 実装仕様
```
型名: PdfString (Hexadecimal)
内部表現: Vec<u8> (バイト列)
パース規則:
  - ホワイトスペースをスキップ
  - 2文字ずつバイトに変換
  - 奇数桁の場合は末尾0補完
```

### 3.6 名前（Name）オブジェクト

```
/Type
/Subtype
/Hello#20World    => "Hello World" (スペースを含む名前)
/paired#28#29parentheses => "paired()parentheses"
/A#42             => "AB"
```

#### 特殊文字のエスケープ

- `#` の後に2桁の16進数を記述することで任意のバイトを表現可能
- ヌル文字（0x00）を含む名前は避けるべき（ただし仕様上は禁止されていない）
- 名前オブジェクトは **アトミックなシンボル** であり、文字列ではない

#### 実装上の注意点
- `/` の直後から次のホワイトスペースまたはデリミタまでが名前の範囲
- 名前は大文字小文字を**区別する**（`/Type` と `/type` は異なる名前）
- 名前の最大長に仕様上の制限はないが、127バイトを推奨上限とする

#### 実装仕様
```
型名: PdfName
内部表現: String (UTF-8デコード後)
パース規則:
  - '/' を検出したらスキャン開始
  - '#' を検出したら次の2文字を16進数としてデコード
  - ホワイトスペースまたはデリミタで終了
比較: バイト列の完全一致
ハッシュ: バイト列ベース
```

### 3.7 配列（Array）

```
[549 3.14 false (Ralph) /SomeName]
```

#### 特徴
- 角括弧 `[` `]` で囲まれた**順序付きのオブジェクトリスト**
- 要素間のデリミタは不要（ホワイトスペースで区切る）
- **異なるデータ型を混在可能**
- **任意の深さでネスト可能**
- 空の配列 `[]` も有効

#### 実装仕様
```
型名: PdfArray
内部表現: Vec<PdfObject>
アクセス: インデックスベース (0始まり)
操作:
  - get(index) -> Option<&PdfObject>
  - push(object)
  - len() -> usize
  - iter() -> Iterator<&PdfObject>
```

### 3.8 辞書（Dictionary）

```
<< /Type /Example
   /Subtype /DictionaryExample
   /Version 0.01
   /IntegerItem 12
   /StringItem (a string)
   /Subdictionary << /Item1 0.4
                     /Item2 true
                     /LastItem (not!)
                  >>
>>
```

#### 制約事項
- 二重の山括弧 `<<` `>>` で囲まれる
- **キーは必ず名前（Name）オブジェクト**でなければならない
- 値には**任意のデータ型**（配列や別の辞書を含む）を指定可能
- 同一キーが複数存在する場合の動作は未定義（最後の値を採用することが多い）
- PDFにおける**最も重要なデータ構造**

#### 実装仕様
```
型名: PdfDictionary
内部表現: HashMap<PdfName, PdfObject> または BTreeMap<PdfName, PdfObject>
操作:
  - get(key: &PdfName) -> Option<&PdfObject>
  - insert(key: PdfName, value: PdfObject)
  - remove(key: &PdfName) -> Option<PdfObject>
  - contains_key(key: &PdfName) -> bool
  - keys() -> Iterator<&PdfName>
  - iter() -> Iterator<(&PdfName, &PdfObject)>
挿入順序: BTreeMapまたはIndexMapで保持を推奨（出力の安定性のため）
```

### 3.9 ストリーム（Stream）

```
<< /Length 44 >>
stream
BT
/F1 24 Tf
72 720 Td
(Hello World) Tj
ET
endstream
```

#### 構造
1. **辞書部分**: ストリームのメタデータを定義
2. **`stream` キーワード**: データ開始を示す（直後にCR+LFまたはLFが必須）
3. **バイナリデータ**: 任意のバイト列
4. **`endstream` キーワード**: データ終了を示す（直前にEOLが必須）

#### 必須辞書エントリ

| キー | 型 | 説明 |
|:-----|:---|:-----|
| `/Length` | 整数 | ストリームデータの**正確なバイト数**（`stream`の後のEOLから`endstream`の前のEOLまで） |

#### オプション辞書エントリ

| キー | 型 | 説明 |
|:-----|:---|:-----|
| `/Filter` | 名前 or 配列 | 適用されている圧縮/エンコードフィルタ |
| `/DecodeParms` | 辞書 or 配列 | フィルタのパラメータ |
| `/DL` | 整数 | デコード後のデータ長（ヒント情報） |

#### 実装仕様
```
型名: PdfStream
内部表現:
  - dictionary: PdfDictionary (メタデータ)
  - data: Vec<u8> (生のバイナリデータ)
  - decoded_data: Option<Vec<u8>> (デコード済みキャッシュ)
操作:
  - decode() -> Result<Vec<u8>> (フィルタを適用してデコード)
  - encode(filter: Filter) -> Result<()> (フィルタを適用してエンコード)
  - raw_data() -> &[u8]
  - dictionary() -> &PdfDictionary
```

## 4. ダイレクトオブジェクトとインダイレクトオブジェクト

### 4.1 ダイレクトオブジェクト

配列内や辞書内に直接インラインで記述されるオブジェクト。

```
[1 2 3]                          % 配列内のダイレクトオブジェクト
<< /Key (value) >>               % 辞書内のダイレクトオブジェクト
```

### 4.2 インダイレクト（間接）オブジェクト

一意の識別子を付与され、ファイル内の任意の場所から参照可能なオブジェクト。

#### 定義構文
```
<オブジェクト番号> <世代番号> obj
  <オブジェクトの内容>
endobj
```

#### 参照構文（インダイレクト参照）
```
<オブジェクト番号> <世代番号> R
```

#### 具体例
```
% オブジェクト番号12、世代番号0の辞書オブジェクト定義
12 0 obj
<< /Type /Page
   /Contents 15 0 R    % オブジェクト番号15、世代番号0への参照
>>
endobj

% オブジェクト番号15、世代番号0のストリームオブジェクト定義
15 0 obj
<< /Length 44 >>
stream
BT /F1 24 Tf 72 720 Td (Hello) Tj ET
endstream
endobj
```

### 4.3 間接参照のメカニズム

- `N G R` は「オブジェクト番号N、世代番号Gのオブジェクトをここで展開せよ」というポインタ
- データの重複を排除する（同じフォントや画像を複数ページで共有）
- 参照先が存在しない場合は `null` として解決される

### 4.4 オブジェクト識別子の構成

| 要素 | 説明 |
|:-----|:-----|
| オブジェクト番号 | 正の整数。ファイル内でオブジェクトを一意に識別 |
| 世代番号 | 非負の整数（通常は0）。オブジェクトが削除・再利用される際にインクリメント |

#### 実装仕様
```
型名: PdfObjectId
内部表現:
  - object_number: u32
  - generation_number: u16
比較: object_numberとgeneration_numberの両方が一致
ハッシュ: (object_number, generation_number) のタプルハッシュ

型名: PdfIndirectRef
内部表現: PdfObjectId
解決: ObjectResolver.resolve(ref) -> Result<PdfObject>
```

### 4.5 キャッシュ戦略

- 解決済みのインダイレクト参照はメモリにキャッシュする
- 巨大なPDFではLRU（Least Recently Used）キャッシュの実装を推奨
- 循環参照の検出機構を必ず実装する（訪問済みオブジェクトIDの追跡）

## 5. コメント

```
% これはコメントです（行末まで有効）
%PDF-1.7                         % ファイルヘッダ（特殊なコメント）
%%EOF                            % ファイル終端マーカー（特殊なコメント）
```

- `%` から行末までがコメントとして扱われる
- コメントはトークンとして認識されず、ホワイトスペースと同等に扱われる
- 例外: `%PDF-` と `%%EOF` は特別な意味を持つ
- ストリームデータ内の `%` はコメントとして解釈されない
