# PDF仕様入門 — pdfmod開発者向けガイド

> **対象読者**: PDFの内部仕様に馴染みのない開発者
> **目的**: pdfmodのコードベースを理解するために必要なPDF仕様の背景知識を提供する
> **参照規格**: ISO 32000-1:2008 (PDF 1.7) / ISO 32000-2:2020 (PDF 2.0)

---

## 1. PDFファイルの全体像

### 1.1 4つのセグメント

PDFファイルは物理的に4つのセグメントで構成されます。

```
+-----------------------------+
| Header                      |  %PDF-1.7（バージョン宣言）
+-----------------------------+
| Body                        |  オブジェクトの集合体
|   1 0 obj ... endobj        |  （ページ、フォント、画像 etc.）
|   2 0 obj ... endobj        |
|   ...                       |
+-----------------------------+
| XRef Table                  |  各オブジェクトのバイト位置を記録した索引
+-----------------------------+
| Trailer + startxref + %%EOF |  ドキュメントのエントリーポイント
+-----------------------------+
```

### 1.2 末尾から読む — PDFの最大の特徴

PDFは**ファイル末尾から逆方向に解析を開始**します。先頭から順に読むのではありません。

```
解析の流れ:

  1. ファイル末尾の %%EOF を見つける
  2. その上の startxref からXRefテーブルのオフセットを得る
  3. XRefテーブルで全オブジェクトの位置を把握する
  4. Trailer辞書の /Root からドキュメント全体にアクセスする
```

この設計により、PDFはファイル全体を読み込まなくても特定のページだけを高速に表示できます。

### 1.3 座標系

PDFの座標原点は**左下**です（数学の座標系と同じ）。1単位 = 1/72インチ（= 1ポイント）。

```
(0, 792) ─────────── (612, 792)
    |                      |
    |     A4サイズの例       |
    |     612 x 792 pt      |
    |                       |
(0, 0) ───────────── (612, 0)
  原点
```

## 2. PDFオブジェクト — 9つの基本データ型

PDFには9種類のプリミティブデータ型があります（ISO 32000 7.3）。
pdfmodでは `PdfObject` 共用体型としてこれらを表現します。

### 2.1 一覧

| # | 型名 | PDF内の記述例 | 説明 | PdfObject の type |
|:--|:-----|:-------------|:-----|:-----------------|
| 1 | Null | `null` | 値が存在しないことを示す | `"null"` |
| 2 | Boolean | `true` / `false` | 論理値 | `"boolean"` |
| 3 | Integer | `42` / `-3` | 整数値 | `"integer"` |
| 4 | Real | `3.14` / `-0.5` | 浮動小数点数 | `"real"` |
| 5 | String | `(Hello)` / `<48656C6C6F>` | 文字列 | `"string"` |
| 6 | Name | `/Type` / `/Pages` | 識別子（辞書のキー等） | `"name"` |
| 7 | Array | `[1 2 3]` | 順序付きリスト | `"array"` |
| 8 | Dictionary | `<< /Key /Value >>` | キー→値のマップ | `"dictionary"` |
| 9 | Stream | `<< /Length 44 >> stream ... endstream` | 辞書 + バイナリデータ | `"stream"` |

### 2.2 各型の詳細

#### Null

```
null
```

「値がない」を明示的に表す型。解決できない間接参照の結果としても返されます。

#### Boolean

```
true
false
```

大文字小文字を区別します（`True` や `FALSE` は無効）。

#### Integer / Real

```
42      ← Integer
-3      ← Integer
3.14    ← Real
-.002   ← Real
```

PDFには「数値」という単一型はなく、整数と実数を区別します。小数点の有無で判定します。

#### String

2つの表記方法があります:

```
(Hello World)        ← リテラル文字列（丸括弧で囲む）
<48656C6C6F>         ← 16進文字列（山括弧で囲む）
```

内部的にはバイト列（`Uint8Array`）として保持します。テキスト抽出には別途 `/ToUnicode` CMap が必要です。

リテラル文字列は `\n`（改行）、`\(`（エスケープされた括弧）、`\053`（8進数）などのエスケープシーケンスを含むことがあります。

#### Name

```
/Type
/Pages
/Font
```

`/` で始まる識別子。辞書のキーとして主に使われます。値としても使われます（例: `/Type /Catalog` の `/Catalog` は値）。

#### Array

```
[1 2.5 (text) /Name [nested array]]
```

任意の `PdfObject` を要素に持てる順序付きリスト。異なる型を混在できます。

#### Dictionary

```
<< /Type /Catalog
   /Pages 2 0 R
   /Version /1.7
>>
```

Name をキー、任意の `PdfObject` を値とするマップ。PDFの中核的なデータ構造で、ページ、フォント、画像など、ほぼすべての構造が辞書で表現されます。

#### Stream

```
4 0 obj
<< /Length 44
   /Filter /FlateDecode
>>
stream
BT /F1 24 Tf 72 720 Td (Hello World) Tj ET
endstream
endobj
```

辞書（メタデータ）+ バイナリデータの組み合わせ。ページの描画命令、画像データ、フォントデータ、さらにはXRefテーブル（PDF 1.5+）もストリームで格納されます。

`/Filter` でデータの圧縮形式を指定します。最も一般的なのは `/FlateDecode`（zlib圧縮）です。

## 3. 間接参照 — オブジェクト同士を繋ぐポインタ

### 3.1 仕組み

PDFのオブジェクトは**間接オブジェクト**として定義され、他のオブジェクトから**間接参照**で指し示されます。

```
定義側（Body内）:
  2 0 obj                     ← オブジェクト番号2, 世代番号0
  << /Type /Pages ... >>
  endobj

参照側（別のオブジェクト内）:
  << /Pages 2 0 R >>          ← 「2 0 R」= オブジェクト番号2, 世代番号0 を参照
```

`N G R` の3つのトークンで構成されます:
- **N**: オブジェクト番号（何番目のオブジェクトか）
- **G**: 世代番号（オブジェクトの更新回数、通常は0）
- **R**: 参照を示すキーワード

### 3.2 世代番号とは

オブジェクトが削除されて再利用される場合、世代番号がインクリメントされます。実務的にはほとんどのPDFで世代番号は `0` です。

### 3.3 pdfmodでの表現

```typescript
// 間接参照: PdfObject の "indirect-ref" バリアント
{ type: "indirect-ref", objectNumber: 2, generationNumber: 0 }

// オブジェクト識別子: ObjectId（IndirectRef のエイリアス）
{ objectNumber: 2, generationNumber: 0 }
```

ObjectResolver（後続Issueで実装）が間接参照を実体のPdfObjectに解決します。

## 4. XRefテーブル — オブジェクトの住所録

### 4.1 役割

XRef（Cross-Reference）テーブルは**「オブジェクト番号 → ファイル内の位置」を記録した索引**です。

PDFリーダーは目的のオブジェクトを読むとき、XRefテーブルからバイトオフセットを取得し、そこにシークしてパースします。これにより巨大なPDFでも必要なオブジェクトだけを効率的に読めます。

### 4.2 テキスト形式（従来形式、PDF 1.4以前）

```
xref
0 6                            ← オブジェクト0から6個のエントリ
0000000000 65535 f             ← オブジェクト0: 空きエントリ (f = free)
0000000009 00000 n             ← オブジェクト1: バイトオフセット9 (n = in-use)
0000000058 00000 n             ← オブジェクト2: バイトオフセット58
0000000115 00000 n             ← オブジェクト3: バイトオフセット115
0000000206 00000 n
0000000300 00000 n
```

各エントリは厳密に**固定長20バイト**:

```
nnnnnnnnnn ggggg s EOL
|          |     |  |
|          |     |  +-- 行末 (CR+LF 等で2バイト)
|          |     +----- 状態フラグ: n=使用中, f=空き
|          +----------- 世代番号 (5桁、0埋め)
+---------------------- バイトオフセット or 次の空きオブジェクト番号 (10桁、0埋め)
```

### 4.3 ストリーム形式（PDF 1.5+）

PDF 1.5からは、XRefテーブルをバイナリのストリームオブジェクトとして格納できるようになりました。テキスト形式より圧縮効率が高く、**オブジェクトストリーム**（type=2）にも対応しています。

```
15 0 obj
<< /Type /XRef
   /Size 20
   /W [1 3 1]           ← 各フィールドのバイト幅: type=1バイト, field2=3バイト, field3=1バイト
   /Root 1 0 R
   /Filter /FlateDecode
>>
stream
... (Flate圧縮されたバイナリデータ) ...
endstream
endobj
```

### 4.4 XRefEntryの3つのタイプ

ストリーム形式では、各エントリは3つのフィールド（`type`, `field2`, `field3`）で表現されます:

| type | 意味 | field2 | field3 |
|:-----|:-----|:-------|:-------|
| **0** | 空きエントリ（削除済み） | 次の空きオブジェクト番号 | 再利用時の世代番号 |
| **1** | 通常のオブジェクト | ファイル内バイトオフセット | 世代番号 |
| **2** | オブジェクトストリーム内 | 親ストリームのオブジェクト番号 | ストリーム内インデックス |

pdfmodではテキスト形式もこの3フィールド形式に正規化して `XRefEntry` 型で統一管理します:

```typescript
interface XRefEntry {
  type: 0 | 1 | 2;
  field2: number;
  field3: number;
}
```

### 4.5 オブジェクトストリーム（type=2）

PDF 1.5で導入された仕組みで、複数の小さなオブジェクトを**1つのストリームにまとめて圧縮**します。

```
通常:                          オブジェクトストリーム:
  1 0 obj ... endobj            10 0 obj
  2 0 obj ... endobj            << /Type /ObjStm /N 3 /First 15 >>
  3 0 obj ... endobj            stream
                                1 0 2 10 3 20    ← ヘッダ（番号 オフセット のペア）
                                << /Type /Page >> ← オブジェクト1の中身
                                [1 2 3]           ← オブジェクト2の中身
                                (text)            ← オブジェクト3の中身
                                endstream
                                endobj
```

XRefEntry の `type=2` はこの形式を指します。`field2` が親ストリームのオブジェクト番号、`field3` がストリーム内のインデックスです。

### 4.6 XRefTable

```typescript
interface XRefTable {
  entries: Map<number, XRefEntry>;  // オブジェクト番号 → エントリ
  size: number;                      // 最大オブジェクト番号 + 1
}
```

## 5. トレイラ辞書 — ドキュメントへの入口

### 5.1 役割

トレイラ辞書はPDF解析の**起点**です。ここから `/Root`（ドキュメントカタログ）を辿ることで、全ページ・全リソースにアクセスできます。

### 5.2 テキスト形式のトレイラ

```
trailer
<< /Size 6
   /Root 1 0 R
   /Info 5 0 R
   /ID [<abc123def456> <abc123def456>]
>>
startxref
408
%%EOF
```

### 5.3 主要エントリ

| キー | 型 | 必須 | 説明 |
|:-----|:---|:-----|:-----|
| `/Root` | 間接参照 | **必須** | ドキュメントカタログへの参照。ここがドキュメント全体のルート |
| `/Size` | 整数 | **必須** | XRefテーブルのエントリ総数（最大オブジェクト番号 + 1） |
| `/Prev` | 整数 | 条件付き | 前のXRefテーブルのバイトオフセット |
| `/Info` | 間接参照 | 任意 | ドキュメント情報辞書（作成者、タイトル等） |
| `/ID` | 配列 | 条件付き | ファイル識別子 `[永続ID, 変更ID]` |

### 5.4 pdfmodでの表現

```typescript
interface TrailerDict {
  root: IndirectRef;              // /Root（必須）
  size: number;                   // /Size（必須）
  prev?: number;                  // /Prev
  info?: IndirectRef;             // /Info
  id?: [Uint8Array, Uint8Array];  // /ID [永続ID, 変更ID]
}
```

## 6. インクリメンタルアップデート — PDFの編集方法

### 6.1 仕組み

PDFは編集時に**ファイル末尾に変更差分を追記**します。元のデータは一切変更しません。

```
+-----------------------------+
| 元のHeader                   |
| 元のBody                     |
| 元のXRef                     |
| 元のTrailer (/Size 10)       |
+-----------------------------+ ← 元のファイルここまで
| 追加・変更されたオブジェクト    |  ← 1回目の編集で追記
| 新しいXRef                   |
| 新しいTrailer                |
|   /Prev 408  ← 前のXRefを指す |
+-----------------------------+
| さらに追加されたオブジェクト    |  ← 2回目の編集で追記
| 最新のXRef                   |
| 最新のTrailer                |
|   /Prev 1024 ← 1回目を指す   |
+-----------------------------+
```

### 6.2 /Prev チェーンとXRefマージ

最新のトレイラの `/Prev` を辿ると前のXRefテーブルに到達します。これを再帰的に辿り、**新しいエントリが古いエントリを上書きする形でマージ**します。

```
最新XRef (優先度: 高)
    ↓ /Prev
1回目XRef (優先度: 中)
    ↓ /Prev
元のXRef (優先度: 低)
    ↓
マージ済みXRefTable
```

同じオブジェクト番号のエントリが複数のXRefに存在する場合、最新のものが勝ちます。

## 7. 全体の解析フロー

```
PDFバイナリ (Uint8Array)
    │
    ▼
[1] %%EOF を末尾から検索
    │
    ▼
[2] startxref の値を取得（XRefテーブルのオフセット）
    │
    ▼
[3] XRefテーブルを解析
    │  テキスト形式 or ストリーム形式を自動判定
    │
    ▼
[4] Trailer辞書を解析
    │  /Prev があれば [3] に戻って再帰的にマージ
    │
    ▼
[5] マージ済みXRefTable + TrailerDict が完成
    │
    ▼
[6] ObjectResolver で /Root を解決
    │  間接参照 → XRefからオフセット取得 → パース → LRUキャッシュ
    │
    ▼
[7] ドキュメントカタログ → ページツリー → 各ページ
    │
    ▼
PdfDocument / PdfPage
```

pdfmodのモジュールとの対応:

| ステップ | pdfmodモジュール | 状態 |
|:---------|:----------------|:-----|
| [1]-[2] | `StartXRefScanner` | 未実装 |
| [3] | `XRefTableParser` / `XRefStreamParser` | 未実装 |
| [4] | `TrailerParser` / `XRefMerger` | 未実装 |
| [5] | 型定義: `XRefTable`, `TrailerDict` | **Issue #7（本Issue）** |
| [6] | `ObjectResolver` + `LRUCache` | LRUCache は **Issue #7**、Resolver は未実装 |
| [6] | `ObjectParser`（Token → PdfObject） | 型定義は **Issue #7**、Parser は未実装 |
| [7] | `PageTreeWalker` | 未実装 |

## 8. 用語集

| 用語 | 説明 |
|:-----|:-----|
| **間接オブジェクト** | `N G obj ... endobj` で定義されるオブジェクト。番号で参照可能 |
| **間接参照** | `N G R` 形式のポインタ。ObjectResolverが実体に解決する |
| **オブジェクト番号** | オブジェクトの一意な識別番号（正の整数） |
| **世代番号** | オブジェクトの更新回数（通常0。削除→再利用でインクリメント） |
| **XRef** | Cross-Reference。オブジェクト番号→ファイル内位置の索引 |
| **トレイラ** | ドキュメントのエントリーポイント。/Root からドキュメント全体にアクセス |
| **インクリメンタルアップデート** | ファイル末尾に差分を追記する編集方式 |
| **オブジェクトストリーム** | 複数のオブジェクトを1つのストリームにまとめる圧縮手法（PDF 1.5+） |
| **FlateDecode** | zlib圧縮。PDFで最も一般的なストリーム圧縮フィルタ |
| **MediaBox** | ページの物理的なサイズを定義する矩形 `[x1 y1 x2 y2]` |
| **CMap / ToUnicode** | 文字コード→Unicode変換マップ。テキスト抽出に必要 |
| **RPN（逆ポーランド記法）** | コンテンツストリームの命令形式。`72 720 Td` = 座標(72,720)に移動 |
